# Рефакторинг колонки «Действия» в таблице заявок

## Контекст

Текущая страница «Заявки» дублирует управление «Исполнителем» и «Услугой»: эти данные есть в отдельных колонках таблицы, но редактируются через select'ы в колонке «Действия». В результате ячейка «Действия» содержит до 5 элементов (Открыть, Взять в работу, Исполнитель, Услуга, Назначить), таблица перегружена и на десктопе выходит за пределы экрана.

## Цель

- Убрать дублирование: редактирование «Исполнителя» и «Услуги» происходит только в своих колонках.
- В колонке «Действия» оставить одну основную кнопку, зависящую от статуса заявки.
- Второстепенные действия спрятать в компактное меню «⋯».
- Сохранить существующую бизнес-логику и права доступа.

## Целевая структура строки таблицы (desktop)

| ID | Корпус | Описание | Услуга (select) | Статус | Создатель | Исполнитель (select) | Срок | Продления | Действия |

## Backend-изменения

### 1. Расширение схемы `RequestUpdate`

Добавить необязательные поля:

- `service_id: Optional[int] = None`
- `assigned_to: Optional[int] = None`

### 2. Расширение `PUT /requests/{request_id}`

В `backend/app/routers/requests.py` в функции `update_request` добавить обработку новых полей:

- Если `data.service_id is not None`:
  - Найти услугу по id с `is_active=True`.
  - Если не найдена — 400.
  - Записать в `req.service_id`.
- Если `data.assigned_to is not None`:
  - Найти пользователя по id с `is_active=True`.
  - Если не найден или роль не входит в `('contractor', 'director', 'admin')` — 400.
  - Записать в `req.assigned_to`.
- Остальные проверки оставить без изменений:
  - Заявка не должна быть в статусе `completed`.
  - `description` и `building_id` обновляются как сейчас.

## Frontend-изменения

### 1. `frontend/src/pages/RequestsListPage.jsx`

#### Колонка «Услуга»

- Для пользователей с `canAssign` (`director`, `admin`) рендерить `<select>` со списком активных услуг.
- При изменении (`onChange`) сразу вызывать `requestsAPI.update(requestId, { service_id })` и обновлять локальный список заявок.
- Для остальных ролей показывать текст `req.service?.name || '—'`.

#### Колонка «Исполнитель»

- Для пользователей с `canAssign` рендерить `<select>` со списком активных пользователей с ролями `contractor`, `director`, `admin`.
- При изменении сразу вызывать `requestsAPI.update(requestId, { assigned_to })` и обновлять локальный список.
- Для остальных ролей показывать текст `req.executor?.full_name || req.executor?.username || '—'`.

#### Колонка «Действия»

Состоит из двух элементов:

1. **Основная кнопка** (`PrimaryActionButton`):
   - `status === 'new' && canTake` → «Взять в работу», вызывает `requestsAPI.take(requestId)`.
   - `status === 'in_progress' && canComplete(req)` → «Завершить», вызывает `requestsAPI.complete(requestId)`.
   - `status === 'completed'` → не рендерить кнопку.

2. **Меню «⋯»** (`ActionsMenu`):
   - Кнопка-иконка с `aria-label="Ещё действия"`, `aria-expanded={isOpen}`.
   - Выпадающий список с `role="menu"`, пункты с `role="menuitem"`.
   - Закрывается по клику вне меню и по нажатию `Escape`.
   - Пункты:
     - **Открыть** — переход `/requests/${req.id}`.
     - **Назначить** — доступно `canAssign` при `status === 'new'`, вызывает `requestsAPI.assign(req.id, req.assigned_to, req.service_id)`. Подтверждает текущие значения из колонок и переводит заявку в `in_progress`.
     - **Печать** — доступно `canPrint`, вызывает `requestsAPI.print([req.id])` и скачивает ZIP с печатной формой одной заявки.
     - **Продлить** — доступно `canExtend(req)`, вызывает `requestsAPI.extend(req.id)`.

#### Удалить из `renderActions`

- Ссылку «Открыть».
- Select'ы «Исполнитель» и «Услуга».
- Кнопку «Назначить» как отдельный элемент.
- Кнопку «Продлить» из основного ряда.
- Кнопку «Завершить» из основного ряда (она станет основной кнопкой).

Функцию `renderActions` можно заменить на inline-рендеринг или сильно упростить.

### 2. Стили

- Ячейка «Действия»: `textAlign: 'right'`, `whiteSpace: 'nowrap'`, `width: '1%'`.
- Контейнер кнопок: `display: 'inline-flex'`, `alignItems: 'center'`, `gap: '6px'`.
- Кнопки и иконка меню: единая высота 32px.
- Выпадающее меню: абсолютное позиционирование, `right: 0`, `zIndex: 10`, белый фон, тень, скругление 8px, минимальная ширина 160px.
- Пункты меню: полная ширина, hover-фон `#f3f4f6`.

### 3. Мобильная версия

- В карточке заявки поля «Услуга» и «Исполнитель» также заменить на select'ы для `canAssign`.
- Блок действий в карточке отрисовывать тем же компонентом `PrimaryActionButton` + `ActionsMenu`.

## Права доступа

- `canAssign`: `director`, `admin` — могут редактировать колонки «Услуга»/«Исполнитель» и видеть пункт «Назначить».
- `canTake`: `contractor`, `director`, `admin` — могут взять новую заявку в работу.
- `canComplete(req)`: `director`/`admin` для любой `in_progress` заявки; `contractor` только если он является исполнителем.
- `canExtend(req)`: только `admin` для незавершённой заявки.
- `canPrint`: `contractor`, `director`, `admin`.

## Чек-лист проверки

- [ ] В ячейке «Действия» максимум 2 видимых элемента (основная кнопка + иконка меню).
- [ ] «Исполнитель» и «Услуга» редактируются только в своих колонках.
- [ ] Основная кнопка меняется по статусу заявки.
- [ ] Меню открывается/закрывается корректно, доступно с клавиатуры, имеет `aria-label`.
- [ ] Ничего не переносится на вторую строку в ячейке «Действия» при обычной ширине окна.
- [ ] Сохранена вся текущая бизнес-логика.
- [ ] Backend-валидация не позволяет назначить неактивного пользователя или неактивную услугу.
- [ ] Завершённую заявку нельзя отредактировать.
