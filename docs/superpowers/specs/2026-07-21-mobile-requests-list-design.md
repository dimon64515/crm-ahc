# Дизайн: мобильный вид списка заявок

## Контекст

Страница `/requests` на мобильных устройствах отображается в виде таблицы с горизонтальной прокруткой. Кнопки «Назначить» и «Открыть» находятся в конце строки, и до них неудобно долистывать.

## Цель

Сделать список заявок удобным на экранах ≤768 px: все действия должны быть видны сразу, без горизонтальной прокрутки.

## Решение

### 1. Breakpoint и layout

- При `window.innerWidth <= 768` показывать карточки вместо таблицы.
- Десктопная таблица остаётся без изменений.

### 2. Структура карточки

Каждая карточка вертикальная, белый фон, скруглённые углы, тень:

1. **Шапка:** ID заявки, цветной бейдж статуса, название корпуса.
2. **Тело:** поля «Описание», «Услуга», «Создатель», «Исполнитель», «Срок» с серыми подписями.
3. **Действия:** flex-контейнер с переносом (`flex-wrap`), в котором располагаются кнопки «Открыть», «Взять в работу», select'ы исполнителя/услуги, кнопка «Назначить», «Продлить», «Завершить».

Кнопки показываются контекстно в зависимости от статуса заявки и роли пользователя (текущая логика `renderActions` сохраняется).

### 3. Сортировка по умолчанию

Backend endpoint списка заявок (`GET /api/requests`) сортирует результат:

- сначала заявки без исполнителя (`executor_id IS NULL`);
- затем заявки с исполнителем;
- внутри каждой группы — по `created_at` убыванию (новые сверху).

### 4. Выделение просроченных заявок

На фронтенде вычисляется признак просрочки: `due_date < сегодня` и статус не `completed`.

- В таблице: ячейка со сроком подсвечивается красным.
- В карточке: поле «Срок» отображается красным цветом с текстом «Просрочено».
