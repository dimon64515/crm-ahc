# Бэкапы и архивирование

## Общее описание

Система поддерживает создание бэкапов двух типов:
1. **Бэкап фото** — архивация фотографий с фильтрами
2. **Полный бэкап** — дамп БД + файлы + справочники

**Доступ:** Только пользователи с ролью `admin`

**Раздел:** `/settings/backups`

---

## 1. Бэкап фотографий

### Назначение

Возможность скачать все фото или выборку по фильтрам одним или несколькими архивами.

### Фильтры

| Фильтр | Тип | Описание |
|--------|-----|----------|
| Период с / по | Date picker | Дата загрузки фото |
| Корпус | Multi-select | Фото из выбранных корпусов |
| Подрядчик | Multi-select | Фото работ выбранных подрядчиков |

### Процесс создания архива

```
[Админ выбирает фильтры] ──► [Система формирует список фото]
                                    │
                                    ├──► Получение путей из БД (work_photos)
                                    ├──► Проверка существования файлов
                                    ├──► Создание ZIP-архива
                                    ├──► Разбиение на части по 300 МБ
                                    └──► Сохранение в uploads/backups/photos/
                                    │
                                    ◄── Ссылки для скачивания
```

### Именование архивов

```
photos_backup_YYYYMMDD_HHMMSS_part{N}.zip

Пример:
  photos_backup_20250527_143022_part1.zip (300 МБ)
  photos_backup_20250527_143022_part2.zip (300 МБ)
  photos_backup_20250527_143022_part3.zip (150 МБ)
```

### Структура внутри архива

```
photos_backup_20250527_143022/
├── metadata.json              # Информация о бэкапе
└── photos/
    ├── building_12/
    │   ├── 20250527_143022_a1b2c3.jpg
    │   └── 20250527_143105_d4e5f6.jpg
    ├── building_15A/
    │   └── 20250526_101200_g7h8i9.jpg
    └── ...
```

### metadata.json

```json
{
  "backup_type": "photos",
  "created_at": "2025-05-27T14:30:22",
  "created_by": "admin",
  "filters": {
    "date_from": "2025-01-01",
    "date_to": "2025-05-27",
    "building_id": [12, 15]
  },
  "total_files": 1250,
  "total_size_mb": 850,
  "buildings": ["12", "15A"]
}
```

## 2. Полный бэкап системы

### Состав

| Компонент | Описание | Имя файла |
|-----------|----------|-----------|
| Дамп БД | SQL-дамп PostgreSQL | `database_dump.sql` |
| Фото | Все фотографии | `photos/` |
| Файлы | Все документы | `files/` |
| Справочник материалов | Экспорт в xlsx | `materials.xlsx` |
| Справочник услуг | Экспорт в xlsx | `services.xlsx` |
| Метаданные | JSON с информацией | `metadata.json` |

### Процесс создания

```
[Админ запускает бэкап] ──► [Система]
                                  │
                                  ├──► pg_dump crm_db > database_dump.sql
                                  ├──► Копирование uploads/photos/
                                  ├──► Копирование uploads/files/
                                  ├──► Экспорт materials → materials.xlsx
                                  ├──► Экспорт services → services.xlsx
                                  ├──► Создание ZIP-архива
                                  ├──► Разбиение на части по 300 МБ
                                  └──► Сохранение в uploads/backups/full/
                                  │
                                  ◄── Ссылки для скачивания
```

### Именование архивов

```
full_backup_YYYYMMDD_HHMMSS_part{N}.zip

Пример:
  full_backup_20250527_143022_part1.zip (300 МБ)
  full_backup_20250527_143022_part2.zip (300 МБ)
  ...
```

### Структура внутри архива

```
full_backup_20250527_143022/
├── metadata.json
├── database_dump.sql
├── materials.xlsx
├── services.xlsx
├── photos/
│   ├── building_12/
│   │   └── ...
│   └── building_15A/
│       └── ...
└── files/
    └── ...
```

### metadata.json

```json
{
  "backup_type": "full",
  "created_at": "2025-05-27T14:30:22",
  "created_by": "admin",
  "version": "1.0.0",
  "database": {
    "name": "crm_db",
    "dump_file": "database_dump.sql"
  },
  "files": {
    "photos_count": 1250,
    "files_count": 340
  },
  "catalogs": {
    "materials_count": 500,
    "services_count": 150
  }
}
```

## 3. Восстановление из бэкапа

### Процесс восстановления

```
[Админ загружает части архива] ──► [Система объединяет части]
                                          │
                                          ├──► Распаковка ZIP
                                          ├──► Валидация структуры (проверка metadata.json)
                                          ├──► [Если full]:
                                          │     ├──► psql < database_dump.sql
                                          │     ├──► Копирование photos/
                                          │     └──► Копирование files/
                                          ├──► [Если photos]:
                                          │     └──► Копирование photos/
                                          └──► Обновление справочников (опционально)
                                          │
                                          ◄── Отчёт о восстановлении
```

### Валидация перед восстановлением

| Проверка | Описание |
|----------|----------|
| Целостность архива | Все части загружены? |
| Структура | Есть metadata.json и database_dump.sql? |
| Версия совместимости | Версия бэкапа совместима с текущей версией системы? |
| Предупреждение | Показать предупреждение: "Все текущие данные будут заменены" |

### Откат (опционально)

Перед восстановлением система автоматически создаёт резервную копию текущего состояния:

```
pre_restore_backup_YYYYMMDD_HHMMSS/
```

## 4. Сжатие фото при загрузке

### Алгоритм

```python
from PIL import Image
import io

def compress_photo(image_bytes: bytes, max_size_mb: float = 1.0) -> bytes:
    """
    Сжимает фото до заданного размера.
    """
    max_size_bytes = max_size_mb * 1024 * 1024
    
    img = Image.open(io.BytesIO(image_bytes))
    
    # Конвертация в RGB (для PNG с прозрачностью)
    if img.mode in ('RGBA', 'P'):
        img = img.convert('RGB')
    
    # Уменьшение размеров (max 1920px по длинной стороне)
    max_dimension = 1920
    if max(img.size) > max_dimension:
        ratio = max_dimension / max(img.size)
        new_size = (int(img.size[0] * ratio), int(img.size[1] * ratio))
        img = img.resize(new_size, Image.LANCZOS)
    
    # Подбор качества для достижения целевого размера
    quality = 95
    output = io.BytesIO()
    
    while quality > 50:
        output.seek(0)
        output.truncate()
        img.save(output, format='JPEG', quality=quality, optimize=True)
        
        if output.tell() <= max_size_bytes:
            break
        
        quality -= 5
    
    return output.getvalue()
```

### Параметры сжатия

| Параметр | Значение |
|----------|----------|
| Максимальный размер | 1 МБ |
| Максимальное разрешение | 1920px по длинной стороне |
| Формат выхода | JPEG |
| Качество (начальное) | 95% |
| Качество (минимальное) | 50% |
| Метод ресайза | LANCZOS |

### Примеры

| Исходный размер | Разрешение | Результат |
|-----------------|------------|-----------|
| 8 МБ | 4000×3000 | ~800 КБ, 1920×1440 |
| 3 МБ | 2500×1800 | ~600 КБ, 1920×1382 |
| 500 КБ | 1280×960 | Без изменений (уже < 1 МБ) |

## 5. Управление бэкапами

### Хранение

```
uploads/backups/
├── full/
│   ├── full_backup_20250527_143022_part1.zip
│   ├── full_backup_20250527_143022_part2.zip
│   └── ...
└── photos/
    ├── photos_backup_20250527_143022_part1.zip
    └── ...
```

### Автоматическая очистка

- Хранить последние **10 полных бэкапов**
- Хранить последние **20 бэкапов фото**
- Старые автоматически удаляются

### Ручное удаление

Администратор может удалить любой архив через интерфейс (`DELETE /api/backups/{backup_id}`).
