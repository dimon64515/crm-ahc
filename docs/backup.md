# Бэкапы и восстановление

## Общее описание

Система поддерживает создание бэкапов двух типов:

1. **Полный бэкап** — дамп PostgreSQL + файлы из `uploads/` (фото, документы, фото заявок).
2. **Бэкап фото** — архив фотографий работ и заявок с фильтрами.

**Доступ:**
- Создание и скачивание архива фото — `director`, `admin` (раздел `/photo-backup`).
- Полный бэкап, управление архивами и загрузка бэкапа для восстановления — только `admin` (раздел `/settings/backups`).

## Полный бэкап

### Состав архива

```
full_YYYYMMDD_HHMMSS/
├── metadata.json
├── database_dump.sql
├── photos/
├── files/
├── works/
└── request_photos/
```

- `metadata.json` — тип, дата, версия, создатель, статистика (количество фото, файлов, материалов, услуг).
- `database_dump.sql` — результат `pg_dump`.
- `photos/`, `files/`, `works/`, `request_photos/` — копии соответствующих поддиректорий `uploads/`.

### Хранение и именование

Архив разбивается на части по `BACKUP_MAX_PART_SIZE_MB` (по умолчанию 300 МБ):

```
full_20260721_120000.part1.zip
full_20260721_120000.part2.zip
...
```

Файлы хранятся в `uploads/backups/`.

### Автоочистка

- Хранятся последние `BACKUP_RETENTION_FULL` полных бэкапов (по умолчанию 10).
- Хранятся последние `BACKUP_RETENTION_PHOTOS` фото-бэкапов (по умолчанию 20).
- Старые бэкапы удаляются автоматически при создании нового.

## Бэкап фото

### Фильтры

- Период (дата работ).
- Корпуса (multi-select).
- Подрядчики (multi-select).

### Состав архива

```
photos_YYYYMMDD_HHMMSS/
├── metadata.json
├── photos/
└── request_photos/
```

## Восстановление

Восстановление выполняется **вручную на сервере** через management-скрипт. Веб-интерфейс позволяет только загрузить бэкап и проверить его целостность.

### Пошаговая инструкция

1. **Создайте или загрузите бэкап.**
   - В `/settings/backups` нажмите «Создать полный бэкап».
   - Или загрузите ZIP-часть бэкапа в раздел «Восстановление из бэкапа» и нажмите «Проверить бэкап».

2. **Проверьте бэкап.**
   После успешной проверки интерфейс покажет команду для восстановления.

3. **Выполните восстановление на сервере.**

   ```bash
   cd /home/dimon64515/projects/crm/backend
   source venv/bin/activate
   python scripts/restore_backup.py --backup-id full_20260721_120000 --yes
   ```

   Скрипт автоматически:
   - создаст pre-backup текущего состояния;
   - остановит backend;
   - пересоздаст базу данных и восстановит её из `database_dump.sql`;
   - заменит файлы в `uploads/` на файлы из архива;
   - запустит backend.

4. **Проверьте работу приложения.**
   Если что-то пошло не так, можно откатиться из pre-backup, созданного скриптом в `uploads/backups/pre_restore_YYYYMMDD_HHMMSS/`.

### Требования

- Скрипт запускается от пользователя с правами на управление systemd-сервисом `crm-backend` и доступом к PostgreSQL.
- Для `psql`, `dropdb`, `createdb` используются параметры из `DATABASE_URL`.

## Конфигурация

Переменные окружения в `backend/.env`:

| Переменная | Описание | Значение по умолчанию |
|------------|----------|----------------------|
| `BACKUP_VERSION` | Версия формата бэкапа | `1.0` |
| `BACKUP_MAX_PART_SIZE_MB` | Максимальный размер одной части архива | `300` |
| `BACKUP_RETENTION_FULL` | Сколько полных бэкапов хранить | `10` |
| `BACKUP_RETENTION_PHOTOS` | Сколько фото-бэкапов хранить | `20` |
